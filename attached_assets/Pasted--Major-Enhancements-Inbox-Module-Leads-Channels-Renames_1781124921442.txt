## 3. Module-driven leads (the biggest new piece)

A **module-driven lead** is a lead created when a patient initiates a transaction through one of the hospital's service modules. These leads carry a **module-specific lifecycle stage** in addition to the general lead status, and they appear automatically in the unified inbox so the agent can act (call / WhatsApp / update stage).

### 3.1 Four module-driven lead types

**Medicine Order**
- Lifecycle stages: `order_received` → `acknowledged` → `confirmed` → `bill_raised` → `payment_received` → `packed` → `out_for_delivery` → `delivered`
- Alternative terminal: `cancelled`
- Source pill: "Medicine Order" (teal/cyan), pill icon

**Lab Test**
- Lifecycle stages: `test_booked` → `sample_collection_scheduled` → `sample_collected` → `in_testing` → `report_ready` → `report_delivered`
- Alternative terminal: `cancelled`
- Source pill: "Lab Test" (amber), flask icon

**Web Appointment Booking**
- Lifecycle stages: `booked` → `confirmed` → `reminder_sent` → `arrived` → `consultation_done`
- Alternative terminals: `cancelled`, `no_show`
- Source pill: "Web Appointment" (blue), calendar icon

**App Appointment Booking**
- Same lifecycle as Web Appointment Booking, source pill "App Appointment" (green), calendar icon
- Source distinguishes for attribution analytics

### 3.2 Auto-creation in the inbox

**AC 3.1 — Module event → inbox lead.** *Given* a transaction event occurs in one of these modules (medicine order placed, lab test booked, appointment booked via web or app), *then* a lead is created or merged (by UHID/mobile) and appears in the unified inbox **automatically, within 5 seconds**.

**AC 3.2 — Source attribution preserved.** The lead's source channel is set to the originating module (`medicine_order`, `lab_test`, `web_appointment`, `app_appointment`). The lead row shows the source pill alongside the general status pill.

**AC 3.3 — Linked transaction record.** Every module-driven lead has a `transactionContext` object capturing module-specific details:
- **Medicine Order:** order ID, list of items (name, qty, price), total amount, delivery address.
- **Lab Test:** booking ID, list of tests, requested date/time, collection address (home collection y/n).
- **Appointment Booking:** appointment ID, doctor, specialisation, appointment date/time, mode (in-person / tele).

**AC 3.4 — Seed data.** On first load, seed **at least 2 leads per module type** (so 2 Medicine Orders, 2 Lab Tests, 2 Web Appointments, 2 App Appointments), spread across different stages so the stepper UI is demoable across stages.

### 3.3 Lifecycle stepper in the lead profile

**AC 3.5 — Vertical stepper renders the module's lifecycle.** *Given* I open a module-driven lead, *then* the right rail (or a dedicated section in the conversation area) shows a **vertical stepper** with every stage in order:
- Completed stages: solid green dot with checkmark, timestamp, actor.
- Current stage: pulsing blue dot, label in bold.
- Future stages: outlined grey dot.
- Alternative terminals (cancelled / no_show): shown as a side-branch with red styling, only one terminal can be active.

**AC 3.6 — Advance to next stage.** Below the stepper, an **"Advance to next stage"** button moves the lead one step forward. Clicking opens a small dialog: "Move from `acknowledged` to `confirmed`? (Optional note)" → Confirm. The stepper updates instantly.

**AC 3.7 — Jump non-sequentially.** A "..." menu beside the button offers **"Jump to stage…"** opening a stage picker. Jumping non-sequentially shows a stronger confirmation ("This skips 2 intermediate stages. Continue?") and logs the skip with reason.

**AC 3.8 — Terminate (cancel).** A second secondary action **"Mark as cancelled"** (or "Mark as no-show" for appointments) closes the lead with a required reason text input. The general status is set to `closed`.

**AC 3.9 — Auto-fulfilment.** *Given* the lead reaches its terminal forward stage (`delivered` / `report_delivered` / `consultation_done`), *then* the general status auto-advances to `fulfilled` and the activity log records `Auto-fulfilled on reaching terminal stage`.

**AC 3.10 — Activity log captures every stage transition** with actor + timestamp + optional note: `[Priya Sharma] Advanced from acknowledged → confirmed at 14:22 · "Patient confirmed delivery address"`.

### 3.4 Transaction context in the lead profile

**AC 3.11 — Module-specific detail card.** Above (or beside) the stepper, render a **Transaction Context card** showing the relevant fields per module:

- **Medicine Order:** Order ID · Items table (name · qty · ₹) · Total ₹ · Delivery address · Payment status (auto-derived from stage: unpaid until `payment_received`, then paid)
- **Lab Test:** Booking ID · Tests list · Scheduled date/time · Collection mode (home/centre) · Collection address (if home)
- **Appointment Booking:** Appointment ID · Doctor · Specialisation · Date/time · Mode

**AC 3.12** These fields are read-only in the agent view (changes happen at the source module). A small "View in [module]" link deep-jumps to the order/test/appointment detail page in the source module.

### 3.5 General status vs. module stage

**AC 3.13 — Two parallel lifecycles.** Module-driven leads have BOTH a general status (`new` → `contacted` → `in_progress` → `fulfilled` / `closed`) AND a module stage. The general status reflects the **agent workflow** (whether the agent has acted); the module stage reflects the **transaction state**. They evolve independently, except:
- When module stage reaches a terminal forward stage, general status → `fulfilled` (AC 3.9).
- When module stage is `cancelled` / `no_show`, general status → `closed`.

**AC 3.14 — Display both.** The lead row shows the general status pill on one line and the module stage pill on a second small line (so they don't conflict). The dashboard "Lead Status Breakdown" continues to chart the general status.

## 4. Composer — universal across channels

The composer adapts to the lead's channel. All composers support **Template insertion**, **Free-text**, **Internal Note** mode (from prior addendum), and a **Call** button.

### 4.1 WhatsApp composer (existing, reaffirm)

**AC 4.1** Free-text within 24h session; template-only when session expired. The "Use template" button is labelled with icon + text (per prior addendum).

### 4.2 SMS — no inbox conversation

**AC 4.2** SMS is **not** in the inbox channel list. SMS is an outbound-only channel for campaigns. The conversation panel never expects SMS replies. If any seed/test data has SMS-source leads, reclassify them.

### 4.3 Email composer (new)

**AC 4.3 — Email composer appears for Email and Form-with-email leads.** Fields:
- **To** — pre-filled with the patient's email (locked).
- **Subject** — required text input. When inserting a template, the subject auto-fills from the template's subject field (templates for the Email channel must include a subject — extend the Template Library schema accordingly).
- **Body** — plain text (rich text is Phase 2). Free-text allowed.
- **Insert template** button — opens the template picker filtered to Email templates.
- **Send** button.

**AC 4.4 — Sent emails appear as outbound bubbles in the thread** with subject and body, channel pill "Email". Mock delivery state ticks (sent → delivered) drive on a short timer.

**AC 4.5 — Email templates in the Library.** Extend the Template Library to support an **Email** channel (in addition to WhatsApp / SMS / Push). Email templates include: name, channel (Email), subject template, body template, goal, language, variables + fallbacks. Seed at least 3 Email templates (Appointment Reminder, Lab Report Ready, Win-back).

**AC 4.6 — Email is now selectable in the Campaign Builder Step 2** as a channel option, with its own per-message rate (seed: ₹0.05) and live preview (mailbox-style preview).

### 4.4 Call button (new, universal)

**AC 4.7 — Call button on every lead with a mobile number.** Visible in the conversation header and on the lead profile. Disabled with tooltip if no mobile is on file.

**AC 4.8 — Mock call modal flow.**
1. Click **Call** → modal opens: "Calling [Patient name] at +91 9876543210…" with a dialing animation (1–2s).
2. Three outcome buttons: **Connected** · **No answer** · **Wrong number**.
3. *Given* "Connected" is clicked, *then* a timer starts (mock "Call duration: 00:12 …"). A **"End call"** button appears.
4. On end-call (or after picking No answer / Wrong number): a small note input ("Outcome notes — optional") + **Save call log**.
5. Save → modal closes. Activity log records: `[Priya Sharma] Called +91 9876543210 · outcome: Connected · duration 1m 12s · "Patient agreed to reschedule"`.

**AC 4.9 — Call from a module-driven lead** logs the same way and is associated with the lead's current stage (so "Called during acknowledged" appears as context in the log).

**AC 4.10 — Multiple consecutive call attempts** are each logged individually. A small "Call attempts: 3" counter appears in the lead profile.

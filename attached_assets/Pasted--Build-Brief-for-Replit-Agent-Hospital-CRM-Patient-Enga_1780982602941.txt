# Build Brief for Replit Agent — Hospital CRM & Patient Engagement Platform (Phase 1 prototype)

Paste this whole brief into Replit Agent. It describes a **functional, clickable prototype** — not a production system. Every external vendor (NotifyVisitors, Karix, PayU/Easebuzz, Meta, DLT) must be **simulated in code**. No real API keys, no real messages, no real payments.

---

## 1. What we're building (one paragraph)

A single web platform for hospitals that (a) captures patient **leads** from every channel into one inbox with correct source attribution, and (b) lets non-marketer staff run **compliant, template-only marketing campaigns** over SMS / WhatsApp / Push. A lead is captured → attributed to a channel → worked through a lifecycle → grouped into a **segment** → engaged via a **campaign**, with the outcome attributed back to the originating lead and channel. It ships as three commercial units sharing one services layer (segments, channels, compliance gate, wallet, integrations): **Lead Capture (CRM)**, **Engagement (self-serve marketing)**, and the **Full Platform** (both, connected end to end). Build the Full Platform; make the two halves visibly separable in the nav.

Brand owner is **Affordplan**. Currency is **INR (₹)**. Timezone is **IST**. The product is for the Indian hospital market and is governed by **TRAI DLT** (SMS) and **Meta HSM** (WhatsApp) rules.

---

## 2. Prototype scope & constraints (read before building)

- **Single hospital tenant** seeded with demo data — no real multi-tenant auth needed.
- **No real login.** Add a persistent **role switcher** in the top bar so a demo viewer can instantly become any role (see §5). Permissions/visibility must actually change with the role.
- **Mock all integrations** (§12). Outbound "sends" produce simulated delivery/open/click events on a timer. Inbound WhatsApp chats are simulated. Wallet top-ups are simulated.
- **Persist data** across page reloads (a simple server-side store or SQLite is fine). Seed it on first boot; add a **"Reset demo data"** button in Settings.
- Keep the **Phase 2 list (§14) out of scope.** Where a Phase 2 channel/feature would appear, show it **greyed out / "Coming in Phase 2"** rather than building it.
- This is a **single-step send** product in Phase 1. The campaign builder is a **3-screen wizard**, but it sends **one message per channel, once** — no multi-step journeys, no time gaps, no AI copy, no free-text message bodies anywhere.

---

## 3. Recommended tech stack

- **Frontend:** React + Vite + TypeScript, Tailwind CSS, a clean component set (shadcn/ui-style). `recharts` for charts. `lucide-react` for icons.
- **Backend:** Node + Express (TypeScript). In-memory seeded store is acceptable; SQLite (better-sqlite3 or Drizzle) preferred for persistence.
- **State:** React Query (or simple fetch hooks) against the Express API.
- Single repo, `npm run dev` starts both. If you prefer Next.js full-stack, that's fine — keep the same structure and behaviors.

Aim for a polished, modern SaaS look. UI inspirations per screen are listed inline (HubSpot, Intercom inbox, CleverTap/MoEngage, Practo, Mixpanel funnels, Stripe billing).

## 5. Roles & permission gates (demo role switcher)

Build these four roles. Permissions must actually gate UI actions, not just labels.

| Role | Can do | Cannot do |

| **Front-office / Lead-desk Exec** (primary) | View dashboards & inbox; pick up / be assigned leads; update lead status; upload leads; trigger outreach; reply in unified inbox; **create** campaigns, segments, and template *requests*; view metrics & integrations hub | **Approve** campaigns; add money to wallet; edit message bodies; change frequency cap |
| **Marketing Manager** (reviewer) | Everything the Exec can, **plus** approve campaigns & outreach; **pause** live campaigns; request templates out-of-flow; **add money to wallet**; **view & modify** Integrations Hub | **Edit a message body** (approve/pause only); change frequency cap |

The **"hospital main user"** who approves campaigns = **Marketing Manager**. The Exec submits; the Manager approves. Affordplan Ops handles backend execution.

## 6. Data model (entities & key fields)

Create these tables/collections and relationships. Seed per §11.

- **Hospital** — id, name, branding (logo, primary color), contact, address, simplifiedMode (bool, default true).
- **User** — id, name, email, role (`exec` | `manager` | `ap_admin`), active.
- **ContactVariable** — id, name, description, mandatory (bool), system (bool). Pre-seed: First name, Last name, UHID, Specialization, Last Visit Date, Mobile.
- **Lead** — id, firstName, lastName, mobile, uhid, specialization, **sourceChannel** (`waba` | `web_chat` | `form` | `csv` | `app_booking` | `web_booking` | `push`… ), status (`new` | `contacted` | `in_progress` | `fulfilled` | `closed`), ownerUserId (nullable), sourceListTag (nullable), optedIn (bool), dndListed (bool), createdAt, lastActionAt. **activityLog[]** (timestamped entries: status changes, messages, assignment).
- **Conversation / Message** — for WABA + web-chat leads: messages[] (direction in/out, body, channel, timestamp), session open/closed (WhatsApp 24h rule).
- **Doctor** — id, name, specialization, schedule/availability (simple slots), active.
- **Specialization** — id, name (used as a lead field & doctor mapping).
- **Appointment** — id, leadId, doctorId, specialization, sourceChannel (`app_booking` | `web_booking`), status (`booked` | `confirmed` | `completed` | `cancelled`), datetime.
- **Segment** — id, name, description, source (`affordplan` | `his` | `csv`), conditionTree (for affordplan/his AND/OR groups), memberLeadIds[] / computed count, lastRefreshAt.
- **Template** — id, name, channel (`sms` | `whatsapp` | `push`), goal, language, body (with `{variable}` tokens), media (nullable), status (`approved` | `pending` | `rejected`), scope (`global` | `hospital`, hidden from hospital UI), dltRegisteredBody (SMS), metaStatus (`APPROVED`|`PENDING`|`REJECTED`) + qualityRating (`Green`|`Yellow`|`Red`) for WhatsApp, senderId (SMS) / wabaContact (WA).
- **TemplateRequest** — id, name, channel, message, media, goal, variables+fallbacks, approvalStage (`ap_marketing` | `channel_compliance` | `live` | `rejected`), createdBy, createdAt.
- **Campaign** — id, name, goal, audienceSegmentId, channels[] (each with templateId, perMessageCost, senderId/wabaContact), estimatedRecipients, costBreakdown (channelCost, fee5pct, gst18pct, total), status (`draft` | `submitted` | `approved` | `live` | `paused` | `completed` | `failed`), complianceResult (per-check pass/fail + dropped count), createdBy, approvedBy, createdAt.
- **CampaignMetrics** — campaignId, counts: sent, delivered, opened, clicked, converted (track both unique UHIDs and mobiles), spend, revenueAttributed, per-channel breakdown.
- **Invoice** — id, campaignId, number, amount, gst, pdfUrl (fake), createdAt.
- **Wallet** — hospitalId, balance, transactions[] (top-up, debit, with timestamp + reference).
- **ChannelConfig** — per channel: type (`sms` | `whatsapp` | `push`), status (`live` | `not_configured` | `pending` | `error`), SMS headers[] (value, ownership `affordplan`|`hospital`, isDefault) up to 5, WABA numbers[] (number, profilePic, ownership, isDefault) up to 5, Push workspaceId (Swasth white-label). Editable in Super-Admin; **mirrored read/limited-edit** in the hospital Integrations Hub.
- **SendRules** — sendWindowStart `09:00`, sendWindowEnd `21:00` IST, frequencyCap (default **3 / patient / 24h**), overridable only by AP Admin.

## 8. The two hero flows (make these flawless for the demo)

### Flow A — Lead → Engagement loop
1. A lead arrives from a channel (simulated inbound WhatsApp, a submitted web form, a CSV upload, or an app/website booking) **or** is uploaded.
2. It **lands in the unified inbox** with source attribution and appears on the collective dashboard. De-dupe by UHID/mobile.
3. It's **assigned / picked up**; status set.
4. The Exec acts: **replies in the unified inbox** (call = Phase 2 greyed).
5. Every send passes the **shared compliance gate**; **wallet debited** on valid launch.
6. Status updated; activity logged; qualified leads **grouped into a segment**.
7. Segment **handed to Engagement** → a campaign runs → **outcomes attribute back** to the lead & channel on the CRM dashboard.

### Flow B — Simplified campaign launch
1. Exec logs in (via role switcher), lands on Engagement Dashboard, clicks **New Campaign**.
2. **Step 1:** goal (fixed) + name + segment → **Show count**.
3. **Step 2:** channel(s) + one template each → live preview + per-message cost; set SMS header / WABA contact.
4. **Step 3:** all-in cost + post-send balance → **Submit for review**.
5. **Manager approves** → compliance gate (6 checks) → wallet debited → **Live**, *or* a specific failure is shown.
6. Live metrics stream to the campaign detail view; completion notification sent.
- **Acceptance:** a trained user finishes login→launch in **under 5 minutes**; **no free-text/AI/token editing** anywhere; launch is **blocked unless all 6 checks pass and the wallet covers cost + 5% fee + 18% GST**.

## 9. Compliance & validation gate — simulate deterministically

Run these **6 checks before wallet debit & launch**. Surface them as inline validation states (a checklist that turns green/red on approval). Make pass/fail **deterministic from seed data** so the demo is repeatable, and include at least one seeded campaign that **fails** each major check on purpose.

| Check | Rule | On fail |
|---|---|---|
| **DLT Template Match (SMS)** | Selected SMS template body == its DLT-registered body, byte-for-byte | **Hard-block** |
| **Meta Approval (WhatsApp)** | Template `metaStatus == APPROVED` **and** `qualityRating != Red` | **Hard-block** |
| **Suppression Scrub** | Drop recipients on opt-out + TRAI **DND** list | **Auto-drop** (remove recipients, continue) |
| **Frequency Cap** | recipient's 24h count + this send ≤ cap (default 3) | **Auto-drop** |
| **Wallet Balance** | balance ≥ channel cost + 5% fee + 18% GST | **Hard-block** |
| **Send Window (SMS promo)** | scheduled time within **09:00–21:00 IST** | **Hard-block** |

Channel constraints to encode: **SMS** needs a DLT-registered 6-char sender ID + exact-match body, promo 09:00–21:00 IST only, DND scrubbing mandatory. **WhatsApp** marketing = Meta-approved HSM only; quality rating Red → sending **auto-paused**; 24h session rule. **Push** = Swasth/Affordplan app only, token pre-issued.

Hard-block → show the specific reason and **do not debit**. Auto-drop → silently remove ineligible recipients (show "N recipients dropped"), then proceed.

## 10. Wallet & billing math

- All-in cost = `Σ(perMessageCost × recipients per channel)` = **channelCost**; then `fee = 5% × channelCost`; `gst = 18% × (channelCost + fee)`; `total = channelCost + fee + gst`. Show this as a table in Step 3 and on the invoice.
- **Low-wallet banner** appears whenever balance **< ₹5,000**.
- **Block approval/launch** if balance < total (this is the Wallet Balance hard-block).
- On approval: **debit wallet**, create a wallet transaction, and **raise an invoice** (number, amount, GST, fake PDF link) auto-listed in Billing.
- **Add money** simulates PayU/Easebuzz: a modal → "processing" → balance updates + a top-up transaction.

## 12. Mock integration behavior

- **NotifyVisitors** (Push/SMS/WABA outbound + analytics) and **Karix** (SMS gateway + inbound web-chat) → a single internal **`mockMessagingService`**.
